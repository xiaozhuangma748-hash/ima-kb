# 34-知识库（ima-kb）提示词/LLM 调用优化诊断报告

> 审查人：提示词工程师
> 日期：2026-07-23
> 范围：所有 LLM 调用点（RAG 问答、HyDE、子问题分解、重排序、记忆抽取、图谱抽取、人格 prompt）
> 身份说明：当前运行模型 `qclaw/pool-hy3-preview`；项目本身调用 Agnes AI（`agnes-2.0-flash`）

## 一、已直接修复的问题（零风险纯 prompt/噪声文本修改，已 py_compile 验证）

### 1. chain.py SYSTEM_PROMPT 第 7 条与第 1 条自相矛盾 【严重】
- 原文第1条："只使用参考资料中的信息，不要编造或使用外部知识"
- 原文第7条："如果参考资料与问题无关，不要用 [n] 标注引用，直接用自身知识回答即可"
- **矛盾后果**：RAG 的"防幻觉"核心约束被自己推翻，模型在资料不相关时会自由编造。
- 修复：第7条改为"明确说明根据现有资料无法回答，建议入库相关文档"（与第2条一致）。

### 2. chain.py user_prompt 向生成模型注入无语义的检索分数 【中】
- 原文对每个资料块都拼：`相关度：{r.score:.4f} (来源: {r.source})`
- 问题：RRF 分数量级极小（约 0.01–0.05），对生成模型是无意义噪声，还可能诱导模型按分数高低而非相关性组织答案。
- 修复：构造喂给 LLM 的资料块时移除分数行，仅保留 `[n] 来源：标题 + 内容`。

### 3. query_transform.py HyDE prompt 硬编码"殡葬政策"领域 【中】
- 原文：`你是一个殡葬政策知识库助手`
- 问题：本仓库（ima-kb v4.1.0）定位是**通用个人知识库**，与目录名"殡葬项目"是两回事。硬编码殡葬会让 HyDE 对非殡葬资料产生严重领域偏置，降低召回质量。
- 修复：改为领域无关的"知识库检索辅助助手"。

### 4. memory/extractor.py & graph/extractor.py JSON 输出约束偏弱 【中，提升稳定性】
- 两个抽取 prompt 只写"严格输出 JSON，不要任何解释文字"，但调用时未强制 JSON 模式，模型偶发输出 markdown 代码块或前缀文字（虽然后端有正则容错，但兜底解析会丢数据）。
- 修复：在 prompt 末尾增加"只输出一个 JSON 对象，不要 markdown 代码块/前缀/解释；确保合法 JSON"的硬性指令。

## 二、发现但未修改（需你拍板/有风险，建议后续处理）

### 5. 两套并存的问答 prompt 系统，且风格人格 prompt 几乎未被主链路使用 【架构/一致性】
- `core/qa/chain.py` 的 `SYSTEM_PROMPT`（通用严谨型）是 CLI/Web 主 QA 真正用的。
- `core/persona/prompts.py` 的 scholar/warrior/artisan/neutral 四套**人格化** system prompt 仅被 `core/pet/administrator.py`（桌面宠物）调用。
- **问题**：用户通过 CLI/Web 问答时，完全没有"学者/战士/工匠"风格；同名项目下两套 prompt 语义重复、维护成本翻倍。
- 建议：要么让主 QA 也支持风格选择（复用 persona 模板），要么在文档中明确"人格系统仅限桌面宠物"。

### 6. 多轮"历史感知检索"配置已开（HISTORY_AWARE_RETRIEVAL=1），但 RAGChain.ask 的 history 参数**完全没用上** 【功能性 bug】
- `chain.py` 的 `ask()` 接收 `history=`，但内部：query 变换走 `transform_query`（仅 HyDE/分解，不用 history），检索后也没有用 summary 扩展 query。
- 注释还写着"替代旧的启发式 _expand_query"——而 `_expand_query()`（唯一读 history 的函数）已是死代码（仅定义，无任何调用）。
- 后果：`ENABLE_CROSS_SESSION_EXTRACT` 提取的记忆、跨轮上下文，在主 QA 检索阶段没生效；只有"无资料降级成纯对话"那一条路径用了 `self.history`。
- 建议：要么接上历史感知（用最近若干轮 summary 扩展检索 query），要么在文档里说明多轮仅作用于"纯对话降级路径"，避免误导。

### 7. 重排序 prompt 用纯 user role、无 system，且未启用 JSON 模式 【健壮性】
- `rerank.py` 直接 `messages=[{"role":"user",...}]`，要求模型返回 JSON 数组。没有 system 角色、没有 JSON 模式指令，靠正则兜底解析（已有 `_parse_scores` 三级兜底）。
- 建议：加一个精简 system 角色（"你是一个相关性打分器，只返回 JSON"），并视 API 支持开启 `response_format={"type":"json_object"}`。

### 8. 子问题分解触发阈值偏宽松 + 解析兜底可能串味 【中】
- `decompose_query` 仅在 `len(query) >= 12` 触发；但 12 字中文可能只是个简单句，被强行拆成多子问题，反而增加检索噪声和 token 成本。
- `_parse_sub_queries` 第3步"换行分隔"兜底：只要 >=2 行就当子问题，可能把模型的一句解释拆成"子问题"。
- 建议：阈值提高到 16–20 字或加"含疑问词/连词才分解"的启发式；收紧兜底解析。

### 9. 桌面宠物人格 prompt 把"检索资料"放进 system，但不给引用编号说明 → 引用混乱风险 【中】
- `persona/prompts.py` 的四套模板都含 `{retrieved_context}` 且要求"关键事实必须引用 [n]"，但模板里**没有告诉模型 [n] 编号与资料块的对应规则**（而 chain.py 的 user_prompt 里明确写了 `[1] 来源：...`）。
- 后果：宠物模式下的引用编号可能错位，与 `citation_validator` 的越界清洗逻辑脱节。
- 建议：在人格模板的"引用规则"里补一句"引用编号 [n] 对应上方资料块 [n]"。

### 10. 生成温度散乱 【一致性，低优先】
- RAG 生成 0.2、纯对话降级 0.5、宠物 0.3、agent 0.3、reader 0.2/0.3、image 0.5。
- 事实性问答（RAG/抽取）建议统一低温度（0.0–0.2），创意类（image/pet 闲聊）可放宽。当前 RAG 0.2 可接受，纯对话降级 0.5 偏高（易跑题/幻觉），建议降到 0.3。
